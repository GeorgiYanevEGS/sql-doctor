| `bitmap_heap_lossy` | `Bitmap Heap Scan` | The Bitmap Heap Scan is operating in lossy mode: work_mem was too small to hold the exact bitmap, so PostgreSQL fell back to tracking which heap pages contain matching rows rather than which exact rows match. Every row on a lossy page must be fetched from the heap and rechecked against the condition — rows that fail the recheck are wasted heap reads. When rows_removed_by_recheck / (actual_rows + rows_removed_by_recheck) exceeds 0.30, more than 30% of all heap reads are wasted. At 0.60+ most of the I/O budget is consumed by recheck failures; the plan is materially degraded by the lossy fallback. Root cause: the bitmap for this query's index condition exceeded the work_mem budget during execution. |
| `cte_scan_single_ref` | `CTE Scan` | A CTE (WITH clause subquery) was materialized and scanned exactly once, reading at least 10,000 rows from the intermediate result. PostgreSQL 12+ inlines non-recursive, side-effect-free CTEs referenced exactly once by default — the presence of a CTE Scan in the plan means the planner chose to materialize instead of inline, adding the cost of writing the full result to a tuplestore and then reading it back. Common causes: the CTE body contains side effects (INSERT/UPDATE/DELETE RETURNING), an explicit MATERIALIZED keyword, or a query structure that prevented inlining. |
| `function_scan_bad_estimate` | `Function Scan` | The planner's row estimate for this set-returning function was badly wrong (actual rows diverged from the estimate by 10x or more). PostgreSQL defaults to a flat estimate of 1000 rows for set-returning functions unless told otherwise — ANALYZE cannot fix this because pg_statistic does not track function output cardinality. A bad estimate here causes the planner to choose the wrong join strategy for any join that consumes this function's output. |
| `hash_join_build_probe_imbalance` | `Hash Join` | The Hash Join hashed a relation that is at least 5x larger than the probe side. PostgreSQL builds an in-memory hash table from the inner (build) side and probes it row-by-row with the outer (probe) side — so the hash table size is determined by the build side's row count. When the planner picks the wrong side to hash, the hash table is much larger than it needs to be, consuming more work_mem and increasing the risk of spilling to disk. The optimal choice is always to hash the smaller relation. |
| `index_only_scan_heap_fetches` | `Index Only Scan` | The Index Only Scan had to visit the heap for more than half its result rows because the visibility map was not up to date. When heap fetches approach actual row count, the scan provides little benefit over a regular Index Scan — the extra overhead of checking the visibility map is paid without saving the heap I/O it's meant to avoid. |
| `initplan_aggregate_expensive` | whole-plan check | Multiple InitPlan subqueries (non-correlated scalar subqueries evaluated once before the main plan) collectively consumed at least 30% of total query execution time, even though no single InitPlan individually crossed the threshold. Each InitPlan executes exactly once and its result is cached, so together they serialise with each other and with the main plan before any output is produced. When several InitPlans each contribute a moderate fraction of execution time, the aggregate cost can dominate even if no individual subquery appears expensive in isolation. |
| `initplan_expensive` | — | An InitPlan (a non-correlated subquery evaluated once to produce a scalar value — typically a MAX/MIN/COUNT used as a filter or assignment) consumed at least 30% of the total query execution time. Unlike a correlated SubPlan (which re-executes once per outer row), an InitPlan executes exactly once and its result is cached. But a single slow InitPlan is still a bottleneck: it serialises with the rest of the plan, so nothing runs until it completes. Common causes: a full Seq Scan inside the subquery computing an aggregate that a supporting index could answer in microseconds, or a missing filter index on the subquery's WHERE clause. |

This forces a full scan on at least one side of the join and prevents the planner from choosing a Merge Join that would exploit a pre-sorted index. |
